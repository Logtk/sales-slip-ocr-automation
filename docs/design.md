# 設計書 — 売上伝票OCR自動化 × 荷主別ルールエンジン

## 0. このプロジェクトが証明したいこと

「OCRで手入力をなくす」という発想は、ベンダー製品のカタログスペックとしてはありふれている。このプロジェクトが示したいのはその一歩先で、次の2点である。

1. **入力自動化(OCR)は必要条件であって十分条件ではない。** 伝票の読み取り精度を上げても、荷主ごとにバラバラな料金体系・付帯作業ルール・チャーター条件が個別にハードコードされている限り、荷主が増えるたびに運用工数は線形以上に膨らむ。ここを解かない限り「自動化したのにスケールしない」という状態から抜け出せない。
2. **全社的には、最初から多数の荷主が並行稼働している。** 一部拠点(自部署に相当)だけが荷主1社に固定されているため単純な運用で済んでいるが、これは全社では例外的な特殊事例にすぎない。本プロジェクトは、その特殊事例から得られた設計要素(外部マスター化・自動反映)を、全社の実態(多数荷主が並行稼働、全国で日次1万件規模)に合わせて拡張した設計(OCRによる入力自動化 + 荷主別ルールエンジンによる例外処理の構造化)を、実データを一切使わない合成シナリオの上で可視化したものである。

対象読者は、社内外のSCM/物流DX関係者、および運用者自身のキャリア(CLO/SCMコンサルタント志向)を評価する採用担当者を想定する。単なる「OCR導入ダッシュボード」ではなく、業務構造そのものへの提案であることが伝わる設計を優先する。

## 1. 匿名化・安全設計(非交渉事項)

以下はこのプロジェクトにおいて例外を認めない制約である。

- **実際のコスト実績を一切使用しない。** 運用者の所属企業における実際の月額コスト・削減額・RPAライセンス費用等の数値は、そのまま公開すると自社の内部コスト構造の推測材料になり得るため、公開リポジトリのどこにも記載しない。本ドキュメント内のコストモデルは、架空の前提値(単価・件数・時間)から独自に組み立てた合成モデルであり、実数値と意図的に一致させていない。
- **伝票種別の構成比(電子/手書き/荷主仕様/代理店)も、実際の測定値そのものではなく「業界的にありうる構成の一例」として提示する。** 特定企業の実測データとして読めないよう、設計書内で明示する。
- **実在の企業名・製品名・導入事例を一切記載しない。** 着想の背景に一般的な業界動向(OCR×基幹システム連携の自動化トレンド)はあるが、特定ベンダーの製品や特定企業の事例を模倣・引用・比較する記載はしない。
- **荷主IDは `荷主A`〜`荷主F`・`ロングテール(集約)` 等のランダム匿名ラベルとし、実在の荷主・取引先を指さない。** 伝票番号・基幹システム名・拠点名も同様に、実在を想起させる固有名詞は使用しない。
- 上記はすべて `mulberry32` シード付き疑似乱数によって再現可能な合成データとして生成し、コード上も「これはダミーデータである」ことが明示的にわかるようにする(変数名・コメント・README双方で明記)。

## 2. 業務ドメインの整理

### 2.1 伝票の4分類(合成シナリオ)

| 分類 | 想定構成比(架空) | 特徴 |
|---|---|---|
| 電子伝票発行システム | 60% | 荷主側システムで発行され、重量等が既にデータとして連携される。OCR対象外。 |
| 自社手書き伝票 | 20% | ドライバーが集荷時に伝票番号をスキャンするが、正確な重量はデポ帰着後の計測を要するケースが多く、後工程での手書き重量記入が発生する。OCR対象。 |
| 荷主仕様伝票 | 10% | 荷主指定フォーマットの帳票。荷主ごとにレイアウトが異なる。OCR対象。 |
| 代理店伝票 | 10% | 他社代理店が発行するフォーマット。自社フォーマットと異なる。OCR対象。 |

電子伝票(60%)は既に構造化データとして連携されるため、本プロジェクトのOCR自動化対象は残り40%(手書き・荷主仕様・代理店)に絞る。

**全社スケールの実態(架空)**: 全社では一日あたり約10,000件の伝票が全国で入力されている。大半の拠点は複数の荷主を同時に担当しており、荷主ごとに異なる料金体系・付帯作業条件・チャーター条件を都度人手で管理しているのが実態である。この「多数荷主が並行稼働している状態」こそが全社の標準像であり、2.3節で扱う自部署のような単一荷主専任拠点は、全社の中ではむしろ例外的な特殊ケースである。

### 2.2 売上訂正フェーズという構造

集荷時にドライバーが伝票番号をスキャンした時点では、重量が未確定・付帯作業やチャーターの有無が未反映であることが多い。このため、後工程の「売上訂正フェーズ」で以下を確定させ、基幹システム(2000年代前半稼働のレガシーシステムを想定)に反映する必要がある。

- 重量の訂正(手書き伝票の実測重量への置き換え)
- 輸送方法パラメータ(航空便: 初便/昼間便/最終便、トラック便: ルート別)
- 集荷時・配達時チャーターの有無
- 付帯作業の有無・内容

混載運賃(重量建て)は基幹システムのマスターに事前登録が可能だが、付帯作業・チャーターはマスター化できないため、都度の手入力(または準自動化)が必要になる。

**根本要因 — 運賃ロジックの「システム上の住所」がない**: 顧客と合意した**包括運賃(基本レート)は正しく合意され、実際に適用もされている**。問題は、その合意済みレートが2000年代前半稼働のレガシー基幹システムに保持されていないことにある。だから毎回、人手で正しい運賃を再現する「売上訂正フェーズ」が構造的に発生する。つまり **課題は商流(顧客との料金合意)にあるのではなく、合意済みの料金をシステムが表現できないこと** にある。

さらに料金は二層で成り立っている。

- **基本包括(基本レート)**: 顧客と合意済みの包括運賃。標準的に適用される。件数の大半を占める。
- **特別適用**: 案件ごとに個別見積するケース(スポットのチャーター・付帯・特殊対応など)。件数は少ないが、訂正工数はここに集中する。

**基幹(または中間レイヤ)が「基本包括レート + 特別適用」を二層で整理・保持できること** が根治の鍵になる。

### 2.3 自部署における特殊事例

一部拠点(自部署に相当)は荷主が1社に固定されているため、チャーター・付帯作業のルールを外部マスターとして保持し、クエリ+RPAで基幹システムへの売上訂正を自動反映するというシンプルな運用が成立している。これは「荷主が1社だから、例外ルールが1セットで済む」という特殊な条件の上に成立する解決策であり、2.1節で述べた全社の標準像(多数荷主が並行稼働)には当てはまらない、あくまで例外的なケースである。

**本質課題は、この特殊事例が使えない大多数の拠点で、荷主ごとに異なる例外ルールがすでに人手で個別管理されており、荷主数が多い拠点ほど対応工数が積み上がっている、という全社ですでに起きている状態である。** これは将来のリスクではなく、現在すでに全社規模で発生している課題として捉える。

## 3. 本質課題と打ち手 — ルールエンジンは「唯一の答え」ではない

OCRによる読み取り自動化だけでは、荷主ごとに異なる料金体系・付帯作業単価・チャーター条件という「業務側の複雑性」は解消されない。むしろOCRで読み取り件数が増えるほど、下流の例外処理(荷主別ルール適用)がボトルネック化するリスクがある。

**重要な前提**: この課題への打ち手は1つではない。入力自動化(OCR+RPA)は症状(手入力)に効くが、原因(料金の複雑性を保持できない基幹システム)には触れない。打ち手のオプション空間(A:入力自動化 / B:基幹改修・中間レイヤ / C:運賃標準化 ほか)と、それぞれの投資回収を定量比較した分析は [strategy_options.md](strategy_options.md) にまとめている。ダッシュボードのセクション6「経営オプション比較」がその可視化にあたる。

以下で述べる **荷主別ルールエンジン** は、そのオプションのうち **B(基幹改修/中間レイヤ)のストラングラー型サブセット** — レガシー基幹を正本として残したまま前段に料金計算レイヤを足す打ち手 — に位置づけられる。自部署の特殊事例(2.3節)で使われている設計要素(外部マスター化・クエリ+RPAによる自動反映)を、全社の実態(多数荷主が並行稼働)に合わせて拡張したものである。

- **ルールエンジンの構造(二層モデル)**: 2.2節の「基本包括 + 特別適用」を、そのまま二層のデータ構造として持つ。
  - **Layer 1: 基本包括レートマスター** — 荷主(または顧客×レーン)をキーに、合意済みの包括運賃・重量帯別単価・付帯/チャーターの標準適用条件を保持する。**現状は基幹に無く人手で再現している部分**であり、ここをシステム化することが根治の本丸。
  - **Layer 2: 特別適用オーバーライド** — 案件ごとの個別見積・例外(スポットチャーター等)を、Layer 1 に対する差分として保持・適用する。
  - OCRで抽出された伝票データ(荷主ID・重量・付帯有無等)に対し、まず Layer 1 の基本包括レートを自動適用し、特別適用がある案件のみ Layer 2 で上書きする。**訂正フェーズが存在するのは Layer 1 がシステムに無いからであり、Layer 1 を持たせれば大半の伝票は自動確定し、人手は Layer 2(特別適用)の確認に絞られる。**
- **荷主追加時のコスト構造の違い**: ルールエンジンがない場合、新規荷主の追加は個別の例外処理ロジック(あるいは人手のExcel運用)の追加を意味し、荷主数に対して工数が逓増する。ルールエンジンがある場合、追加は「マスターテーブルへの1レコード登録」に近づき、荷主数が増えても運用工数の増加は緩やかになる。この違いをダッシュボードの「荷主別カスタム料金ルールエンジン」画面で、荷主数(1〜60社程度、全社規模を想定した架空レンジ)をパラメータとしたコスト比較曲線として可視化する(架空の合成コストモデルによる)。
- **荷主構成のモデル化**: 実際の3PLによくある「少数の主要荷主が件数の大半を占め、残りは多数の小口荷主(ロングテール)」という構造を反映し、主要荷主6社(架空)+ロングテール集約カテゴリの7区分で提示する。ロングテール部分こそ、荷主ごとの個別対応が最も積み上がりやすい領域であり、ルールエンジンの必要性を最も強く裏付ける。
- **カスタマイズ性は捨てない**: 荷主ごとの料金カスタマイズは、大手キャリアと差別化する自社の付加価値であるという前提に立ち、「カスタマイズを画一化して排除する」のではなく「カスタマイズをルールとして構造化し、運用コストを荷主数に対してスケールする形に変える」ことをゴールとする。

## 4. ダッシュボード構成

`index.html` は以下6画面(セクション)で構成する。

1. **課題サマリー**: 全社スケール(全国日次1万件・架空)の伝票内訳構成比、売上訂正フェーズの課題ステートメント。自部署のような単一荷主専任拠点は「特殊事例」としてケーススタディ・コールアウトボックスで別枠表示する
2. **OCR処理シミュレーション**: 伝票種別ごとの信頼度分布、自動承認/要人手確認の閾値スライダー、抽出結果サンプルテーブル
3. **売上訂正ワークフロー可視化**: OCR抽出→重量確定→輸送方法パラメータ判定→チャーター判定→付帯作業判定→基幹システム登録、のフローとステップ別自動化率
4. **荷主別カスタム料金ルールエンジン**: 荷主別ルール一覧、荷主数スライダーによるルールエンジンあり/なしの工数・コスト比較
5. **ROI/コストシミュレーション**: 荷主数・月間伝票件数をパラメータとした合成コストモデルによる削減額シミュレーション
6. **経営オプション比較**: 打ち手を1つに固定せず、A(入力自動化)/B(基幹改修・中間レイヤ)/C(特別適用の標準化)を回収期間・NPVで比較。Bの改修投資額・Cの標準化削減率スライダーに加え、**土台スライダー D(電子化率=母数)・E(集約=単価)** でベースラインそのものを動かせる。payback曲線・各案の回収月/NPV・C→Bの順序効果(Bを安くする)・D/Eの代替関係(源流で問題を小さくするほど重いBは割に合わなくなる)を可視化。下部に Impact×Effort の2×2ポジショニング(A/B/C/D/E)と、E→D→B の投資順序ロードマップ(安い上流を先に効かせ、重いBは最後に是非を判断)も配置(合成モデル、詳細は [strategy_options.md](strategy_options.md))

## 5. 合成データ設計

### 5.1 OCR抽出結果(伝票単位)

| フィールド | 型 | 説明 |
|---|---|---|
| `slip_id` | string | 匿名連番ID(例: `SL-000123`) |
| `slip_type` | enum | `handwritten` / `shipper_format` / `agency_format` |
| `shipper_id` | string | `荷主A`〜`荷主F`(主要荷主)または `ロングテール(集約)` のいずれか |
| `extracted_weight_kg` | number | OCR抽出重量(合成値) |
| `confidence` | number(0-100) | OCR信頼度スコア(合成値) |
| `has_charter` | boolean | チャーター有無(合成値) |
| `has_incidental_work` | boolean | 付帯作業有無(合成値) |
| `status` | enum | `auto_approved` / `needs_review` |

信頼度スコアは伝票種別ごとに異なる平均・分散を持つ正規分布(0-100にクリップ)から `mulberry32` シード付き乱数で生成する。伝票フォーマットが自社所定でない `agency_format` ほど平均信頼度を低く、`shipper_format` は荷主ごとに整った様式であるため比較的高め、`handwritten` は最も分散が大きい、という3者の差をつけて合成する(いずれも架空の前提値)。

### 5.2 荷主別ルールマスター

| フィールド | 型 | 説明 |
|---|---|---|
| `shipper_id` | string | `荷主A`〜`荷主F`(主要荷主) または `ロングテール(集約)` |
| `weight_tier_rates` | array | 重量帯別単価テーブル(架空) |
| `incidental_rate` | number | 付帯作業単価(架空) |
| `charter_condition` | string | チャーター適用条件(架空、簡略化した文言) |

主要荷主6社が全体件数の大半を占め、残りは多数の小口荷主(ロングテール、集約カテゴリとして1行で表示)という構造で、実際の3PLによくある荷主構成を模している。

### 5.3 合成コストモデルの前提

すべて架空の前提値であり、実際のコスト実績とは意図的に対応させていない。

- 人手処理1件あたりのみなし処理時間・時間単価(架空)
- OCR自動化後の残存人手件数 = 全体件数 × (1 - 自動承認率)
- ルールエンジンなしの月間運用工数(架空、荷主数 n の関数): `4n + 0.15n(n-1)` — 荷主間の例外対応が増えるほど組み合わせ的に工数が積み上がることを表現した多項式
- ルールエンジンありの月間運用工数(架空、荷主数 n の関数): `8 + n` — 固定のオンボーディングコスト+荷主ごとのマスター登録という定型作業のみ
- 荷主数 n は 1〜60 程度のレンジを想定し、全社規模(主要荷主+ロングテール)での運用を模擬する。n=1 が2.3節の特殊事例(自部署相当)に対応する

## 6. 設計原則

- 実OCRエンジンは統合しない。OCR統合の技術的検証よりも、抽出後の業務ロジック(訂正フェーズ・ルールエンジン)の設計力を見せることを優先する。
- 単一 `index.html` + Tailwind CDN + Vanilla JS + SVG というスタックを踏襲し、ビルド不要・GitHub Pages直配信を維持する(既存案件 `Deliv_arrangement` 等との一貫性)。
- 表示用ラベル(日本語)とデータのキー(英語)を分離し、将来的な項目追加や英語圏向け展開があっても表示層だけの変更で対応できるようにする。

## 7. 未確定・要確認事項

- 荷主別ルールマスターの重量帯区分数・単価レンジは、実運用のイメージに近づけつつも完全に架空の値とする。過度にリアルに寄せすぎないよう、実装時に確認する。
- コスト比較曲線の関数形(線形 vs 逓増/逓減カーブの具体的な係数)は、ストーリーとして伝わりやすい形に実装時点で調整する。

## 8. 意図的スコープ外

- 実際のOCRエンジン統合・画像処理・帳票レイアウト解析
- 実際の基幹システムとの連携仕様(API・DB接続等)
- 荷主別ルールのインタラクティブな編集UI(本プロジェクトでは静的可視化+シミュレーションパラメータに留める)
- 複数拠点・複数事業所展開を前提としたマルチテナント設計の詳細
