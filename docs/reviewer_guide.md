# レビューガイド — 立場別の見どころと想定問答

このデモを見る人の立場によって、注目すべきポイントは異なる。4つの視点それぞれについて「見るポイント｜答え｜所在」と想定問答をまとめる。前提として、本プロジェクトの数値はすべて合成データであることを踏まえて読んでほしい([design.md](design.md) 参照)。

## 1. SCM戦略コンサルタント視点

| 見るポイント | 答え | 所在 |
|---|---|---|
| なぜ「OCR導入」ではなく「OCR×ルールエンジン」なのか | OCRは入力自動化(必要条件)にすぎず、荷主ごとの料金カスタマイズという業務側の複雑性は解消しない。ルールエンジンで例外処理を構造化することで初めてスケールする | design.md 3章 |
| カスタマイズ性を「排除」ではなく「構造化」する設計判断 | 大手ではない3PLにとって荷主別カスタマイズは差別化要因であり、画一化はむしろ competitive disadvantage になり得るという業界理解に基づく | design.md 0章・3章 |
| スケーラビリティの定量的根拠 | 荷主数に対する運用工数カーブ(ルールエンジンあり/なし)をシミュレーションで比較。荷主が増えるほど差が拡大する構造を可視化 | index.html セクション4 |
| 打ち手を1つに固定しない姿勢 | OCR+ルールエンジンは有力な一案(オプションB)にすぎない。A(入力自動化)/B(基幹改修)/C(運賃標準化)を回収期間・NPVで並べ、前提次第で最適解が変わることを示す。「答えありき」で構築しない | index.html セクション6、strategy_options.md |

**想定問答**
- Q. 「OCR単体の製品と何が違うのか？」→ A. OCRは前段の自動化にすぎない。本提案の主眼は後段の「荷主別ルールをどう構造化して運用コストをスケールさせるか」で、OCRはその入力チャネルの一つという位置づけ。
- Q. 「なぜ答えを1つに絞らないのか？」→ A. 答えを1つに固定すると、その答えありきの分析になる。issue tree で根本原因を分解し(strategy_options.md 1章)、原因ごとに異なる打ち手をオプションとして並べ、投資回収で比較することで、意思決定者が前提を動かしながら判断できるようにしている。
- Q. 「このシミュレーションの前提値はどこから来ているのか？」→ A. すべて架空の合成値。実企業のコスト構造を推測されないよう意図的にフィクション化しており、根拠は design.md 5章・strategy_options.md 4章に明記している。

## 2. 物流DX・情報システム担当視点

| 見るポイント | 答え | 所在 |
|---|---|---|
| 根本要因は商流でなくシステム | 包括運賃(基本レート)は顧客と合意済み・適用済みだが、レガシー基幹に保持されていない。だから毎回人手で再現する訂正フェーズが発生する。課題は料金合意ではなくシステム側のギャップ | design.md 2.2節、strategy_options.md 1章 |
| 二層レートモデルの設計 | ルールエンジンを Layer1(基本包括マスター)+ Layer2(特別適用オーバーライド)で構成。大半の伝票は Layer1 で自動確定し、人手は少数の特別適用に絞られる | design.md 3章、index.html セクション4 |
| レガシー基幹システムとの連携方針 | 実際のDB/API連携仕様には踏み込まず、あくまで業務ロジック(訂正フェーズの構造)の設計に焦点を当てている(意図的スコープ外) | design.md 8章 |
| OCRエンジンをどう選定するか | 本デモでは実OCRを統合していない。技術検証よりも抽出後の業務ロジック設計力を見せることを優先した設計判断 | design.md 6章「設計原則」 |
| 伝票種別ごとの信頼度設計 | 手書き伝票が最も分散が大きく、荷主仕様伝票は様式が整っているため相対的に信頼度が高い、という3者の差をつけて合成データを設計 | design.md 5.1節、index.html セクション2 |

**想定問答**
- Q. 「実際にどのOCRエンジンを使う想定か？」→ A. 本デモでは特定のエンジンを想定していない。実装フェーズでは信頼度スコアを返せるエンジン(クラウドOCR API等)であれば置き換え可能な設計。
- Q. 「基幹システムへの反映はどう実装するのか？」→ A. 現行踏襲であればクエリ+RPAでの反映が現実的だが、本デモの範囲は「訂正フェーズで何を確定させる必要があるか」の可視化までで、連携実装は範囲外としている。

## 3. 現場売上処理担当者視点

| 見るポイント | 答え | 所在 |
|---|---|---|
| 「要人手確認」に振り分けられる基準 | OCR信頼度スコアが閾値未満の伝票。閾値はスライダーで調整でき、閾値を下げるほど自動承認は増えるが誤りが混入するリスクとのトレードオフになる | index.html セクション2 |
| 売上訂正フェーズで何を確認しているか | 重量・輸送方法パラメータ(航空便/トラック便区分)・チャーター有無・付帯作業有無、の4点 | design.md 2.2節、index.html セクション3 |
| 荷主が増えても自分たちの負担が増えない仕組み | ルールエンジンが荷主ごとの単価・条件をマスターとして保持するため、都度の確認作業ではなく例外時のみの確認に近づく | index.html セクション4 |

**想定問答**
- Q. 「閾値を上げすぎると現場の確認作業が増えないか？」→ A. その通りで、閾値と確認件数はトレードオフの関係にある。ダッシュボードのスライダーはそのトレードオフを体感できるようにするためのもの。

## 4. 経営層(投資判断)視点

| 見るポイント | 答え | 所在 |
|---|---|---|
| 投資対効果の考え方 | 荷主数(1〜60社)・伝票件数をパラメータとした運用工数の比較モデル。荷主1社(自部署のような特殊事例)なら現行運用でも大差ないが、全社規模の荷主数になるほどルールエンジンの投資回収効果が大きくなる | index.html セクション5 |
| 複数の打ち手の回収比較 | A(入力自動化)/B(基幹改修)/C(特別適用の標準化)を回収期間・36ヶ月NPVで比較。Aは速いが浅く、Bは深いが重く、Cは源流に効くが商流交渉を要する。投資余力・時間軸で最適解が変わることをスライダーで体感できる | index.html セクション6、strategy_options.md 4章 |
| 補完と代替の使い分け | 土台スライダー D(電子化=母数)・E(集約=単価)でベースラインを縮小すると、重いBの回収が伸び NPV が負に転じうる。「Cは B を安くする(補完)、D/E は B の必要性を削る(代替)」という役割の違いを定量的に示す | index.html セクション6、strategy_options.md 4.1節 |
| なぜ今この投資が必要なのか | 荷主1社に閉じた運用が成立するのは自部署のような一部拠点の特殊事例のみ(design.md 2.3節)。全社では最初から多数荷主が並行稼働しており、この課題は将来のリスクではなく現在すでに全社規模で発生していることを架空モデルで定量的に示している | design.md 2.1・2.3節 |
| リスク・未確定要素 | ルールマスターの粒度設計やコストモデルの係数は実装時に精査が必要な未確定事項として明記している | design.md 7章 |

**想定問答**
- Q. 「結局どの打ち手が正解か？」→ A. 一つに定めないのが本分析の趣旨。前提(投資額・標準化の進み具合)を動かすと最適解が変わる。ダッシュボードのセクション6はその感度を体感させる道具として設計している。
- Q. 「実際の削減額はいくらか？」→ A. 本デモは実額を意図的に含んでいない。実際の投資判断には、自社の実データに基づく再計算が別途必要になる。

## 5分デモの推奨動線

1. **0:00–0:30** ヘッダーと課題サマリー(セクション1) — 全国日次1万件・多数荷主並行稼働という全社スケールの課題提起と、自部署のような単一荷主専任拠点が特殊事例であることの対比
2. **0:30–1:30** OCR処理シミュレーション(セクション2) — 閾値スライダーを動かし、自動承認率と要人手確認件数のトレードオフを実演
3. **1:30–2:15** 売上訂正ワークフロー(セクション3) — フロー図でステップごとの自動化率の違いを見せる
4. **2:15–3:00** ルールエンジン(セクション4) — 荷主数スライダーを動かし、コスト曲線が分岐していく様子を実演。ここが本プロジェクトの核
5. **3:00–3:30** ROIシミュレーション(セクション5) — セクション4の設定値がそのままROIに反映されることを示す
6. **3:30–4:30** 経営オプション比較(セクション6) — B/C/D/Eスライダーを動かし、payback曲線と各案の回収月・NPVが変化する様子を実演。「答えは一つではない」「Cは Bを安くする(補完)、D/Eは Bの必要性を削る(代替)」を口頭で添え、2×2で「どこから手をつけるか」、ロードマップで「E→D→B の投資順序」を示す。ここが今回の一番の見せ場
7. **4:30–5:00** design.md「0章」「2.2節(根本要因=合意済み運賃がシステムに無い)」と strategy_options.md を開き、設計思想・issue tree・非交渉事項(匿名化方針)を説明して締める
