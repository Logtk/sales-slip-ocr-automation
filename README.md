# Sales_slip_ocr_optim — 売上伝票OCR自動化 × 荷主別カスタム料金ルールエンジン

**「OCRで手入力をなくす」だけでは、荷主ごとにバラバラな料金体系は解決しない。** 全社では一日約10,000件(架空)の伝票が全国で入力され、大半の拠点で多数の荷主が並行稼働している。この規模でスケールする売上訂正フェーズの設計を、合成データのみで可視化したデモです。

> ⚠️ すべてのデータは合成(架空)データです。実在企業のコスト実績・荷主情報は一切含まれていません。詳細は [docs/design.md](docs/design.md) の「1. 匿名化・安全設計(非交渉事項)」を参照してください。

## 課題ストーリー

- 売上伝票は4種類に分かれる: 電子伝票発行システム(荷主連携・OCR対象外)/ 自社手書き伝票 / 荷主仕様伝票 / 代理店伝票(後者3種がOCR自動化の対象)
- 集荷時点では重量・付帯作業・チャーター有無が未確定なことが多く、「売上訂正フェーズ」で確定させて基幹システムへ反映する必要がある
- 全社では最初から多数の荷主(少数の主要荷主+多数のロングテール)が並行稼働しており、荷主ごとに異なる料金体系・付帯作業条件を都度人手で管理しているのが実態
- 荷主数が多い拠点ほど、この個別ルールへの対応工数が積み上がる — **OCRによる入力自動化だけでは解決しない本質課題**
- 荷主ごとのカスタマイズ対応力は、大手キャリアと差別化する自社の付加価値でもあり、単純に排除することもできない
- **ケーススタディ**: 一部拠点(自部署に相当)は荷主が1社に固定されているため、外部マスター+クエリ+RPAというシンプルな運用が例外的に成立している。しかしこれは全社では特殊事例であり、大半の拠点には当てはまらない

この課題に対し、本プロジェクトは「荷主別カスタム料金ルールエンジン」による構造化を提案し、荷主数(1〜60社、全社規模を想定)に対する運用工数の違いを合成モデルでシミュレーションします。

## ダッシュボードの構成

- **構成**: 単一 `index.html`(Tailwind CDN + Vanilla JS + 標準SVG、外部チャートライブラリ不使用、モックデータ動的生成)
- **ローカル確認**: `index.html` をブラウザで開くだけ(Tailwind CDNのためネット接続が必要。ビルド不要)
- **GitHub**: https://github.com/Logtk/sales-slip-ocr-automation
- **デモ**: https://logtk.github.io/sales-slip-ocr-automation/ (`main` 直配信・GitHub Actions不要。pushで数分後に自動反映)

### 画面

1. **課題サマリー** — 伝票内訳の構成比、売上訂正フェーズの課題ステートメント
2. **OCR処理シミュレーション** — 伝票種別ごとのOCR信頼度分布(合成値)、自動承認閾値スライダー、抽出結果サンプルテーブル(フィルタ・ページネーション付き)
3. **売上訂正ワークフロー可視化** — OCR抽出→重量確定→輸送方法判定→チャーター判定→付帯作業判定→基幹システム登録の各ステップと自動化率
4. **荷主別カスタム料金ルールエンジン** — 荷主別ルール一覧、登録荷主数スライダーによるルールエンジンあり/なしの運用工数カーブ比較
5. **ROIシミュレーション** — 荷主数に応じた月間運用工数の比較(すべて架空モデル)
6. **経営オプション比較 — 答えは一つではない** — 打ち手を1つに固定せず、A(入力自動化)/B(基幹改修・中間レイヤ)/C(特別適用の標準化)を回収期間・NPVで比較。B投資額・C標準化率に加え、土台スライダー D(電子化=母数)・E(集約=単価)でベースライン自体を動かせる。C→Bの順序効果(Cは Bを安くする補完)、D/Eの代替関係(源流で問題を小さくするほど重いBは割に合わなくなる)、Impact×Effortの2×2、E→D→Bの投資順序ロードマップまでを一枚で提示

すべてのグラフは `mulberry32` シード付き疑似乱数で再現可能な合成データから描画しています。

## 設計思想

- **打ち手を1つに固定しない。** OCR+ルールエンジンは有力な一案(オプションB)にすぎず、基幹改修・運賃標準化・課金モデル転換など、原因の異なる複数の打ち手を回収期間で比較する。「答えありき」で構築しないことを重視している
- OCRエンジンの実統合はスコープ外とし、抽出後の業務ロジック(売上訂正フェーズ・ルールエンジン)と経営オプションの設計に焦点を当てる
- 実際のコスト実績・荷主情報は一切使用せず、架空の前提値から独自に組み立てた合成モデルのみで構成する
- 表示用ラベル(日本語)とデータのキー(英語)を分離し、将来の項目追加が表示層だけの変更で対応できるようにする

詳細な設計根拠は [docs/design.md](docs/design.md)、経営オプション分析は [docs/strategy_options.md](docs/strategy_options.md)、用語対照表・匿名化方針は [docs/DATA_DICTIONARY.md](docs/DATA_DICTIONARY.md) を参照してください。
